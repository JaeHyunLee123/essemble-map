# Context Notes

* 작성일: 2026-05-06
* 목적: 합주실 지도 앱 초기 설계 단계의 의사결정 기록

## 주요 아키텍처 결정 사항
1. **서버리스 백엔드:** Next.js의 API Routes와 Supabase를 결합하여 서버 운영 비용을 최소화하고 인프라 관리 포인트를 줄임.
2. **모노레포 내 크롤러 배치:** 크롤링 스크립트를 독립된 서비스로 분리하지 않고 `scripts/` 폴더에 포함시켜, Drizzle ORM 스키마의 중복 작성을 방지하고 유지보수를 용이하게 함.
3. **이미지 저장소 일원화:** 유저가 URL을 입력하더라도 백엔드가 해당 이미지를 캡처하여 Supabase Storage에 복사해 둠으로써 외부 트래픽 방어와 영구적인 스토리지 관리를 확보함.
4. **마커 클러스터링 적용:** 지도 렌더링 성능 확보를 위해 Viewport 쿼리 및 프론트엔드 클러스터링 방식을 적극 채택.
5. **유연한 카테고리 설계:** 장비의 종류가 늘어날 것에 대비해 데이터베이스 ENUM 타입 대신 `EquipmentCategories` 테이블을 별도 운용.

14. **네이버 지도 기반 위경도 자동 추출**: 제보 시 유저에게 직접 위도, 경도나 주소를 입력받지 않고 네이버 지도 링크만 제출받도록 설계하여 UX를 단순화함. 백엔드에서 단축 URL 리다이렉트 추적 및 플레이스 요약 API 조회를 수행하여 위경도를 자동으로 추출해 냄. 파싱 실패 시에는 즉시 에러를 반환하여 제보를 반려함.
15. **MVP 스코프 다운 (방/장비 및 이미지 배제)**: 개발 속도와 빠른 릴리즈를 위해 첫 MVP 버전에서는 방(Room)과 장비(Equipment), 이미지 업로드 기능을 구현하지 않음. 단, 상세 조회 API(`GET /api/studios/:id`)는 프론트엔드의 추후 확장성을 위해 빈 방 배열(`rooms: []`)을 포함하여 응답하도록 보존함. 이미지 관련 API 및 Storage 업로드 로직은 전면 보류함.
16. **Next.js 16 Proxy (`proxy.ts`) 기반 라우트 제어 도입**: Next.js 16의 새로운 아키텍처 규격에 따라 기존의 `middleware.ts` 대신 `proxy.ts`를 사용합니다. 마이페이지(`/mypage` 등)와 어드민 페이지(`/admin/*`)에 대한 접근 제어, 권한 검증 및 비인가 유저 리다이렉트 흐름은 별도의 복잡한 프록시 솔루션 대신 `proxy.ts`에서 단일하고 중앙집중화된 방식으로 통합하여 처리하기로 결정했습니다.
17. **유저 북마크의 Active 상태 검증 조건 수립**: 데이터 정합성과 신뢰성 있는 UX를 위해, 북마크 목록 API(`GET /api/user/bookmarks`) 조회 시 DB 이너 조인과 함께 `Studios.status = 'active'` 필터링을 백엔드 단에서 강력하게 강제합니다. 이를 통해 북마크 이후 반려되거나 비활성화된 합주실이 유저 목록에 잘못 노출되는 경우를 원천 차단합니다.
18. **제보 반려 히스토리 및 반려 사유 제공 규칙 수립**: 도메인 규칙을 바탕으로, 유저 제보 내역 API(`GET /api/user/submissions`) 및 UI에서는 승인/대기뿐만 아니라 반려(`deny`)된 제보 히스토리와 그에 대한 구체적인 반려 사유(`denyReason`)를 안전하게 유저에게 제공하기로 설계했습니다.

## 향후 주의사항
* Playwright 기반 크롤러가 Vercel 배포 환경에 불필요하게 포함되지 않도록 빌드 타임 패키징 설정(또는 `devDependencies` 분리)에 신경 쓸 것.
* 네이버 지도 자동 파싱 기능의 경우, 네이버 내부 API의 주소나 규격이 예고 없이 변경될 수 있으므로, 주기적인 작동 여부 모니터링 및 주기적인 예외 처리 코드 정비가 필요함.
24. **Zustand 초기화 실패 시 세션 청소 로직 보강**: 브라우저 로컬스토리지에 만료된 토큰이 기억되어 있는 상태에서 앱 초기화 시 리프레시 토큰 검증(`/api/auth/refresh`)이 실패하는 경우, Zustand 스토어 내에 낡은 세션 데이터를 방치하지 않고 확실하게 `user`와 `accessToken`을 `null`로 초기화하도록 로직을 보강했습니다. 이를 통해 로컬스토리지 세션과 쿠키 유효성 간의 불일치로 인한 어드민 대시보드 무한 로딩 오류를 원천 차단합니다.
